四、主要内容

什么是软件架构？

  • 软件架构被定义为管理以下规则、启发式和模式：
    1. 将问题和系统划分为离散的部分
    2. 用于在这些部分之间创建接口的技术
    3. 用于管理整体结构和流程的技术
    4. 用于将系统与其环境连接起来的技术
    5. 适当使用开发和交付方法、技术和工具

为什么架构很重要？

  • 软件架构的主要目标是
    1. 定义系统的非功能性需求
    2. 并定义环境。
  • 详细设计之后是如何在架构规则中交付功能行为的定义。架构很重要，因为它：
    1. 控制复杂性
    2. 实施最佳实践
    3. 提供一致性和均匀性
    4. 提高可预测性
    5. 支持重用

什么是OOP？

  • OOP是一种设计理念。它代表面向对象编程。Object-O riented Programming （OOP） 使用一组与旧过程编程语言（C、Pascal 等）不同的编程语言。OOP 中的所有内容都被归类为自我可持续的“对象”。因此，您可以通过四个主要的面向对象编程概念获得可重用性。
  • 为了清楚理解面向对象模型，我们以你的“手”为例。“手”是一个类。你的身体有两个“手”类型的物体，分别命名为“左手”和“右手”。它们的主要功能由一组通过肩膀（通过接口）发送的电信号来控制或管理。因此，肩膀是你的身体用来与你的手互动的界面。手是一个架构良好的类。手被重新使用，通过稍微改变它的属性来创建左手和右手。

核心、基础、对象、Class、Object

什么是对象？

  • 对象可以被视为可以执行一组相关活动的“事物”。对象执行的一组活动定义了对象的行为。例如，手（对象）可以抓握某物，或者学生（对象）可以给出他们的名字或地址。
  • 在纯 OOP 术语中，对象是类的实例。

什么是班级（类、Java中的cLass类）？

  • 类只是对象类型的表示形式。它是描述对象详细信息的蓝图、计划或模板。
  • 类是创建各个对象的蓝图（Java中使用类 class 创建出来的实例 new Object()；instance ）。
  • 类由三件事组成：名称（类名）、属性（类的属性）和操作（类的方法method）。
  • 在现实世界中，您经常会发现许多相同的单个对象。例如，可能有数千辆其他自行车存在，所有品牌和型号都相同。每辆自行车都是根据相同的蓝图建造的。在面向对象的术语中，我们说自行车是称为自行车的对象类的一个实例。
  • 在软件世界中，虽然你可能没有意识到这一点，但你已经使用过类了。例如，您始终使用的控件是由定义其外观和功能的类组成的。每次拖动控件时，实际上都在创建该类的新实例。TextBoxTextBoxTextBoxTextBox

如何识别和设计类？

  • 这是一门艺术;每个设计器（Java、C、C++等软件编程语言）都使用不同的技术来识别类。
  • 但是，根据面向对象设计原则，在设计类时必须遵循五个原则：
    1. SRP - 单一责任原则 - 一个班级应该有一个，也只有一个，改变的理由。（类的不可改变）
    2. OCP（开放封闭原则）应该能够扩展任何类的行为，而无需修改类 （类的继承）
    3. LSP - Liskov 替换原则 - 派生类必须可替换其基类。 （子类的属性、方法 覆盖 ）
    4. DIP - 依赖反转原则 - 依赖于抽象，而不是具体 （abstract）
    5. ISP - 接口隔离原则 - 创建特定于客户端的细粒度接口 (interface)
  • 此外，为了正确识别类，您需要标识系统的叶级函数或操作的完整列表（系统的粒度级用例）。然后，您可以继续对每个函数进行分组以形成类（类将对相同类型的函数或操作进行分组）。但是，定义良好的类必须是一组函数的有意义的分组，并且应该支持整个系统的可重用性，同时提高可扩展性或可维护性。
  • 在软件世界中，总是推荐分而治之的概念，如果你一开始就开始分析一个完整的系统，你会发现它更难管理。因此，更好的方法是先确定系统的模块，然后分别深入挖掘每个模块以寻找类。
  • 软件系统可以由许多类组成。当您有很多类时，需要对其进行管理。想想一个大型组织，其员工人数超过几千人（让我们把一个员工作为一个班级）。为了管理这样的劳动力，您需要制定适当的管理政策。同样的技术可以应用于管理软件系统的类。为了管理软件系统的类并降低复杂性，系统设计人员使用了多种技术，这些技术可以分为四个主要概念，分别称为：
    1. 封装
    2. 抽象化
    3. 遗产
    4. 多态性

这些概念是面向对象编程世界的四大神，在软件术语中，它们被称为面向对象编程（OOP）的四个主要概念。

对象的属性、衍生

什么是封装（或信息隐藏）？

  • 封装是在程序对象中包含对象运行所需的所有资源，基本上是方法和数据。在OOP中，封装主要是通过创建类来实现的，类公开了方法和属性。类是一种容器、胶囊或单元格，它封装了一组方法、属性和属性，以向其他类提供其缩进功能。从这个意义上说，封装还允许类在不损害系统整体功能的情况下更改其内部实现。封装的思想是隐藏一个类如何执行其业务，同时允许其他类向它发出请求
  • 为了模块化/定义一个类的功能，该类可以以许多不同的方式使用另一个类公开的函数或属性。根据面向对象编程，类可以使用多种技术相互链接。这些技术被命名为关联、聚合和组合。
  • 封装还有其他几种方式，（使用接口、匿名类）

什么是关联？

  • 关联是两个类之间的 （*a*） 关系。它允许一个对象实例使另一个对象实例代表它执行操作。关联是定义两个类之间关系的更通用的术语，而聚合和组合则相对特殊。
  • 对于一些初学者来说，联想是一个令人困惑的概念。不仅由关联本身造成的麻烦，而且由另外两个 OOP 概念（即关联、聚合和组合）造成的麻烦。在描述聚合和组合之前，每个人都理解关联。聚合或组合不能单独理解。如果你只理解聚合，它就会破解联想的定义，如果你试图单独理解组合，它总是会威胁到聚合的定义，这三个概念都是密切相关的，因此必须通过比较一个定义来研究另一个定义。让我们探索这三个概念，看看我们是否能理解这些有用概念之间的差异。

关联、聚合和组合有什么区别？

  • 关联是两个阶级之间的关系，其中没有特定的所有权。它只是两个类之间的连接。在另一个类中定义一个类的变量时，首先启用以关联第二个类的函数和属性。话又说回来，聚合和组合都是关联的类型。*has-a*
  • 聚合是一种具有部分所有权的弱关联类型。对于聚合关系，我们使用该术语来表示弱关系。与组合相比，这很弱。再说一次，弱意味着聚合器的链接组件可以在聚合生命周期中幸存下来，而没有其父对象的存在。例如，学校部门教师。任何教师都可以属于多个部门。因此，如果一个部门不复存在，老师仍然存在。*uses**has-a**uses*
  • 另一方面，组合是一种具有完全所有权的强关联类型。与弱聚合相比，这是很强的。对于组合关系，我们使用该术语来表示牢固的关系。例如，一个部门课程，这意味着任何课程的生命周期都取决于该部门的生命周期。因此，如果一个部门不复存在，基础课程也将不复存在。*owns**has-a**owns*
  • 每当没有所有权时，我们就把这种关系看作是一种关联，我们只是使用这个术语，或者有时是描述这种关系的动词。例如，老师或学生。教师和学生之间没有所有权，每个人都有自己的生命周期。*has-a**has-a**teaches*
  • 所以总而言之，我们可以说聚合是一种特殊的关联，而组合是一种特殊的聚合。（关联->聚合->组成)

对象的限制、规范、约束

什么是抽象和泛化？（概念）

  • 抽象是强调思想、品质和属性，而不是细节（对细节的压制）。抽象的重要性源于它隐藏不相关细节的能力，以及使用名称来引用对象。抽象在程序的构造中是必不可少的。它强调一个对象是什么或做什么，而不是它如何表示或如何工作。因此，它是管理大型程序复杂性的主要手段。
  • 抽象通过隐藏不相关的细节来降低复杂性，而泛化则通过用单个结构替换执行类似功能的多个实体来降低复杂性。泛化是将应用范围扩大到包含相同或不同类型对象的更大域。编程语言通过变量、参数化、泛型和多态性提供泛化。它强调对象之间的相似性。因此，它通过将个人收集到组中并提供可用于指定组中任何个人的代表来帮助管理复杂性。
  • 抽象和泛化通常一起使用。摘要通过参数化进行推广，以提供更大的实用性。在参数化中，实体的一个或多个部分将替换为该实体的新名称。该名称用作参数。当调用参数化抽象时，它是通过参数绑定到参数来调用的。

什么是抽象类？（编程中的实现）

  • 使用关键字声明的抽象类无法实例化。它只能用作扩展类的其他类的超类。 类是一个概念，当它被子类实现时，实现就完成了。除此之外，一个类只能从一个类继承（但一个类可以实现许多接口），并且必须覆盖其声明为并可能覆盖虚拟方法/属性的所有方法/属性。abstractabstractAbstractabstractabstract

什么是接口？

  • 总之，接口将实现分开并定义了结构，这个概念在需要实现可互换的情况下非常有用。除此之外，当实现频繁更改时，接口非常有用。
  • 接口可用于定义泛型模板，然后使用一个或多个类来定义接口的部分实现。接口只指定方法声明（隐式 和 ），并且可以包含属性（也是隐式 和 ）。接口定义以关键字 interface 开头。无法实例化像类这样的接口

类和接口有什么区别？

  • 类和接口是两种不同的类型（概念上）。从理论上讲，类强调封装的思想，而接口强调抽象的思想（通过抑制实现的细节）。两者构成了彼此之间的明显区别。因此，在两者之间进行有效有意义的比较是非常困难的，或者说是不可能的，但是在接口和抽象类之间进行比较是非常有用和有意义的。

Interface 和 Abstract 类有什么区别？ 接口和抽象类有什么区别？

  • 接口和抽象类之间有很大的区别，尽管两者看起来很相似：
    1. 接口定义以关键字 interface 开头，因此它属于 interface 类型。
    2. 抽象类是用 abstract 关键字声明的，因此它属于类类型。
    3. 接口没有实现，但必须实现。
    4. 抽象类的方法可以有自己的默认实现，并且可以扩展。Abstract 类的方法可以独立于继承类运行。
    5. 接口只能具有方法声明（隐式 和 ）和属性（隐式publicabstractpublic static)
    6. 抽象类的方法不能仅在声明时才有实现。abstract
    7. 接口可以继承多个接口。
    8. 抽象类可以实现多个接口，但只能继承一个类。
    9. 抽象类必须覆盖所有抽象方法，并且可以覆盖虚拟方法。
    10. 当实现发生变化时，可以使用接口。
    11. 抽象类可用于为基类提供一些默认行为。
    12. 接口使实现可互换。
    13. 接口通过隐藏实现来提高安全性。
    14. 在实现框架时可以使用抽象类。
    15. 抽象类是创建计划继承层次结构的绝佳方法，也可以用作类层次结构中的非叶类。
  • 抽象类允许您定义一些行为;他们强迫你的子类提供其他类。例如，如果您有一个应用程序框架，则可以使用抽象类来提供服务的默认实现以及所有必需模块，例如事件日志记录和消息处理等。此方法允许开发人员在框架提供的引导式帮助中开发应用程序。
  • 但是，在实践中，当您遇到一些只有您的应用程序才能执行的特定于应用程序的功能时，例如启动和关闭任务等。抽象基类可以声明虚拟关闭和启动方法。基类知道它需要这些方法，但抽象类让你的类承认它不知道如何执行这些操作;它只知道它必须启动操作。当需要启动时，抽象类可以调用启动方法。当基类调用此方法时，它可以执行子类定义的方法。

什么是隐式和显式接口实现？

  • 多种实现，隐式和显式实现的概念通过隐藏、公开或保留每个接口方法的标识，提供了实现多个接口方法的安全方法，即使方法签名相同也是如此。

对象的复用、更多设计

什么是继承？

  • 专业化、泛化
  • 通过扩展现有类从现有类创建新类的能力称为继承。
  • 新类 （） 称为派生类或子类，它继承了现有类 （） 的成员，该类称为基类或超类。该类可以通过添加新类型和方法以及重写现有类型和方法来扩展类的功能。
  • 就像抽象与泛化密切相关一样，继承与专业化密切相关。重要的是将这两个概念与概括一起讨论，以便更好地理解和降低复杂性。
  • 现实世界中对象之间最重要的关系之一是专业化，可以描述为“is-a”关系。当我们说狗是哺乳动物时，我们的意思是狗是一种特殊的哺乳动物。它具有任何哺乳动物的所有特征（它有年轻的生命，有牛奶的护士，有毛发），但它将这些特征专门用于家犬的熟悉特征。猫也是一种*哺乳动物。因此，我们希望它与哺乳动物中普遍存在的某些特征具有某些特征，但在猫中专门具有的特征有所不同。
  • 专业化和泛化关系既是互惠的，也是分层的。专业化只是概括硬币的另一面：哺乳动物概括了狗和猫之间的共同点，而狗和猫将哺乳动物特化为自己的特定亚型。
  • 在 OOP 中，专用化关系是使用称为继承的原则实现的。这是实现这种关系的最常见、最自然和最被广泛接受的方式。

什么是多态性？

  • 多态性是一个通用术语，意思是“许多形状”。更准确地说，多态性意味着请求由各种不同类型的事物执行相同操作的能力。
  • 有时，我曾经认为理解面向对象编程的概念变得困难，因为它们被归入四个主要概念，而每个概念都彼此密切相关。因此，必须非常小心地分别正确理解每个概念，同时理解每个概念与其他概念的关系。
  • 在 OOP 中，多态性是通过使用许多不同的技术来实现的，这些技术分别称为方法重载、运算符重载和方法重写。

什么是方法重载？

  • 方法重载是定义多个具有相同名称的方法的能力。

什么是运算符重载？

  • 运算符重载（不太通常称为临时多态性）是多态性的一种特殊情况，其中部分或全部运算符（如 +、- 或 ==）被视为多态函数，因此根据其参数的类型具有不同的行为。

什么是方法覆盖？

  • 方法重写是一种语言功能，它允许子类重写由其超类之一提供的方法的特定实现。
  • 子类可以给出自己的方法定义，但需要具有与其超类中的方法相同的签名。这意味着在重写方法时，子类的方法必须与超类的重写方法具有相同的名称和参数列表。

表达设计

什么是用例？

  • 用例是参与者从系统中感知到的东西。用例将参与者与功能映射在一起。重要的是，演员不一定是人。例如，当一个系统与另一个系统通信时，它可以扮演参与者的角色。
  • 从另一个角度来看，用例对用户与系统的典型交互进行编码。具体而言，它：
    1. 捕获一些用户可见的功能。
    2. 为用户实现一些具体的目标。
  • 一整套用例在很大程度上定义了系统的要求：用户可以看到和想要做的一切。

什么是类图？

  • 类图被广泛用于描述系统中对象的类型及其关系。类图使用类、包和对象等设计元素对类结构和内容进行建模。类图描述了设计系统时的三个不同视角：概念、规范和实现。这些透视在创建图表时变得明显，并有助于巩固设计。
  • 类图、物理数据模型以及系统概述图是适合当今快速应用程序开发要求的最重要的图表。
  • UML 表示法

什么是封装图？

  • 包图用于反映包及其元素的组织。当用于表示类元素时，包关系图提供了命名空间的可视化效果。在我的设计中，我使用包图将类组织到系统的不同模块中。

什么是序列图？

  • 序列图以可视化方式对系统内的逻辑流进行建模，它既可以记录和验证逻辑，又可用于分析和设计目的。序列图是动态建模中最流行的 UML 工件，它侧重于识别系统内的行为。

架构思想

什么是两层架构？

  • 两层架构也指客户端/服务器架构，术语客户端/服务器于 1980 年代首次用于指网络上的个人计算机 （PC）。实际的客户端/服务器模型在 1980 年代后期开始被接受，后来它被改编为万维网编程。
  • 根据现代使用的两层架构，用户界面（或 ASP.NET，所有网页）在客户端上运行，数据库存储在服务器上。实际的应用程序逻辑可以在客户端或服务器上运行。因此，在这种情况下，用户界面直接访问数据库。这些也可以是非接口处理引擎，为其他远程/本地系统提供解决方案。无论哪种情况，今天的两层模型都不如三层模型那么有名。两层设计的优点是简单，但简单性伴随着可扩展性的成本。较新的三层体系结构（更为著名）为应用程序逻辑引入了中间层。

什么是三层架构？

  • 三层软件架构（也称为三层架构）出现在 1990 年代，以克服两层架构的局限性。这种架构已被现代系统设计人员积极定制并采用到 Web 系统中。
  • 三层是一种客户端-服务器架构，其中用户界面、功能流程逻辑、数据存储和数据访问作为独立模块进行开发和维护，有时在单独的平台上进行。术语“三层”或“三层”，以及多层架构（通常称为三层架构）的概念，似乎起源于 Rational Software。三层体系结构包含以下三层：
    1. 表示层或 Web 服务器：用户界面，显示/接受数据/输入/来自用户
    2. 应用程序逻辑/业务逻辑/事务层或应用程序服务器：数据验证、添加到数据库之前的可接受性检查以及所有其他特定于业务/应用程序的操作
    3. 数据层或数据库服务器：对数据库或任何其他存储、连接、命令、存储过程等的简单读写方法。

什么是 MVC 架构？

  • Model-V iew-C ontroller （MVC） 架构将基于用户输入的领域建模、表示和操作分离为三个单独的类。
  • 不幸的是，这种模式的流行导致了许多错误的用法;每种技术（Java、ASP.NET 等）都以自己的方式定义它，使其难以理解。特别是，“控制者”一词在不同的上下文中被用来表示不同的事物。下面给出的定义是我为 ASP.NET 版本的 MVC 找到的可能的定义。模型：数据集和类型化*# 数据集（有时是业务对象、对象集合、XML 等）是模型最常见的用法。
    1. 视图：ASPX 和 ASCX 文件通常处理视图的职责。
    2. 控制器：事件的处理或控制通常在代码隐藏类中完成。
  • 在复杂的 n 层分布式系统中，MVC 架构在组织系统的表示层方面发挥着至关重要的作用。

什么是 SOA？

  • 面向服务的体系结构本质上是服务的集合。这些服务相互通信。通信可以涉及简单的数据传递，也可以涉及两个或多个服务协调某些活动。需要一些将服务相互连接的方法。
  • SOA 可以用作连接多个系统以提供服务的概念。它在 IT 世界的未来中占有重要地位。
  • 根据上面的假想图，我们可以看到如何使用面向服务的架构为一个国家的公民提供一组集中式服务。公民将获得一张独特的身份证，该卡带有每个公民的所有个人信息。每个服务中心，如购物中心、医院、车站和工厂都配备了一个计算机系统，该系统连接到一个中央服务器，该服务器负责为城市提供服务。例如，当客户进入购物中心时，区域计算机系统将其报告给中央服务器，并在提供对场所的访问之前获取有关客户的信息。系统欢迎客户光临。客户完成购物，然后当他离开购物中心时，他将被要求完成计费流程，区域计算机系统将管理该流程。付款将使用从客户识别卡获得的输入详细信息自动处理。区域系统将向城市报告（城市的计算机系统），而城市将向国家报告（国家的计算机系统）。

什么是数据访问层？

  • 数据访问层 （DAL） 是每个 n 层系统的关键部分，主要由一组简单的代码组成，这些代码与数据库或任何其他存储设备进行基本交互。这些功能通常称为 CRUD（创建、检索、更新和删除）。
  • 数据访问层需要尽可能通用、简单、快速和高效。它不应包括复杂的应用程序/业务逻辑。
  • 我见过具有冗长、复杂的存储过程 （SP） 的系统，这些过程在进行简单检索之前会运行多种情况。它们不仅包含大部分业务逻辑，还包含应用程序逻辑和用户界面逻辑。如果 SP 变得越来越长、越来越复杂，那么这是一个很好的迹象，表明您正在将业务逻辑埋在数据访问层中。

什么是业务逻辑层？

  • 我知道这是大多数人的问题，但另一方面，通过阅读许多文章，我意识到并不是每个人都同意业务逻辑到底是什么，在许多情况下，它只是表示层和数据访问层之间的桥梁，没有太多， 除了从一个拿走并传递到另一个。在其他一些情况下，甚至没有经过深思熟虑，他们只是从表示层和数据访问层中取出剩余部分，然后将它们放在另一个层中，该层自动称为业务逻辑层。然而，在软件世界中，没有上帝所说的事情是无法改变的。当您觉得您应用的方法足够灵活以支持系统的增长时，您可以进行更改。有很多很好的方法，但在选择它们时要小心，它们可能会使简单的系统过于复杂。这是一个人需要与自己的经验找到的平衡。
  • 一般建议，在定义业务实体时，必须决定如何将表中的数据映射到正确定义的业务实体。业务实体应有意义地定义，考虑系统的各种类型的要求和功能。建议标识业务实体以封装应用程序的功能/UI（用户界面）要求，而不是为数据库的每个表定义单独的业务实体。例如，如果要合并来自几个表的数据以生成 UI（用户界面）控件（Web 控件），请在业务逻辑层中使用使用几个数据对象来支持复杂业务需求的业务对象实现该函数。

什么是四人帮 （GoF） 设计模式？

  • 四人帮 （GoF） 模式通常被认为是所有其他模式的基础。它们分为三类：创造型、结构型和行为型。在这里，您将找到有关这些重要模式的信息。
  • 创作模式
    1. 抽象工厂 - 创建多个类族的实例
    2. 构建器 - 将对象构造与其表示分离
    3. Factory 方法 - 创建多个派生类的实例
    4. 原型 - 要复制或克隆的完全初始化的实例
    5. Singleton - 只能存在单个实例的类
  • 结构模式
    1. 适配器 - 匹配不同类的接口
    2. Bridge - 将对象的接口与其实现分开
    3. 复合对象 - 简单对象和复合对象的树结构
    4. 装饰器 - 动态地向对象添加职责
    5. Facade - 表示整个子系统的单个类
    6. Flyweight - 用于高效共享的细粒度实例
    7. 代理 - 表示另一个对象的对象
  • 行为模式
    1. Chain of resp - 一种在对象链之间传递请求的方法
    2. 命令 - 将命令请求封装为对象
    3. 解释器 - 一种在程序中包含语言元素的方法
    4. 迭代器 - 按顺序访问集合的元素
    5. Mediator - 定义类之间的简化通信
    6. Memento - 捕获并恢复对象的内部状态
    7. 观察者 - 一种通知多个类更改的方法
    8. 状态 - 当对象的状态发生变化时改变对象的行为
    9. 策略 - 将算法封装在类中
    10. 模板方法 - 将算法的确切步骤推迟到子类
    11. Visitor - 定义一个不做任何更改的类的新操作

  • 差异:
    • 1抽象工厂：强调一系列产品对象（简单或复杂）
    • 1构建器：专注于逐步构建复杂对象
    • 2抽象工厂：专注于*什么*被制造出来
    • 2建造者：专注于*如何*制作
    • 3抽象工厂：专注于定义许多不同类型的工厂*来构建许多*产品*，而不是只为一种产品构建一个构建器
    • 3构建器：专注于构建一个复杂但单一的*产品*
    • 4抽象工厂：将要制作的具体对象类型的选择推迟到运行时
    • 4构建器：隐藏如何编译该复杂对象的逻辑/操作
    • 5抽象工厂：*每个*方法调用创建并返回不同的对象
    • 5生成器：只有 *last* 方法调用返回对象，而其他调用部分生成对象
  • 有时，创建模式是互补的：因此，在设计系统时，您可以加入一个或多个模式。例如，构建器可以使用其他模式之一来实现构建哪些组件，或者在另一种情况下，Abstract Factory、Builder 和 Prototype 可以在其实现中使用 Singleton。因此，结论是，这两种设计模式的存在是为了解决两种类型的业务问题，因此即使它们看起来很相似，它们也不是。

5. 结论是什么？

  • 我不认为试图让编程语言成为每个人的一切是不现实的。语言变得臃肿、难以学习、难以阅读，如果所有东西加上厨房水槽都被扔进去。换句话说，每种语言都有其局限性。作为系统架构师和设计者，我们应该能够完全且更重要的是正确地（这也意味着你不应该使用弹道导弹来杀死一只苍蝇或雇用联邦调查局来捕捉苍蝇）利用可用的工具和功能来构建可用、可持续、可维护且非常重要的可扩展软件系统，充分利用该语言的功能为他们的客户带来具有竞争力的先进系统。为了做到这一点，系统的基础起着至关重要的作用。软件系统的设计或架构是基础。它将系统结合在一起，因此正确设计系统（这绝不意味着*过度*设计）是成功的关键。当你谈论设计一个软件系统时，正确处理OOP概念是非常重要的。我使上述文章的思想更加丰富，但仍然保持简短，以便人们可以一目了然地学习/提醒所有重要概念。希望大家会喜欢阅读它。
